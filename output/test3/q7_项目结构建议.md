## 模块职责说明

| 模块 | 职责 | 对应代码来源 |
|------|------|-------------|
| **`src/utils/`** | 通用的、无业务依赖的纯工具函数。无状态、可复用、易测试。 | Q1-Q2 的 `merge_dicts`、`safe_divide`、`get_user` |
| **`src/analytics/`** | 业务数据的统计分析逻辑。封装筛选规则和聚合计算。 | Q6 重构后的 `ActiveRecordAnalyzer` |
| **`src/api/`** | 对外暴露的 API 接口层。负责编排调用、组装响应，不包含具体查询逻辑。 | Q4-Q5 的 `get_dashboard` |
| **`src/db/`** | 数据库访问层（Repository 模式）。封装 SQL 查询，对上层提供类型化的数据访问接口。 | Q5 中提取的 `_fetch_user`、`_fetch_orders` 等 |
| **`src/external/`** | 外部服务调用层。封装 HTTP 客户端、超时策略、降级逻辑、重试机制。 | Q5 中的 `_fetch_notifications` |
| **`tests/unit/`** | 单元测试。每个工具函数/类一个测试文件，覆盖正常/边界/异常路径。 | Q3 的 pytest 用例 |
| **`tests/integration/`** | 集成测试。验证跨模块协作是否正确（如 API → DB → External）。 | dashboard 端到端场景 |
| **`scripts/`** | 运维脚本。不属于应用代码，但与项目运维相关。 | test2-Q6 的安全清理脚本 |

## 设计原则

```
┌─────────────┐
│   api/      │  ← 编排层：调用 db/ 和 external/，组装响应
├─────────────┤
│ analytics/  │  ← 业务逻辑层：纯计算，不依赖外部 I/O
├─────────────┤
│   db/       │  ← 数据访问层：封装 SQL
│ external/   │  ← 外部调用层：封装 HTTP
├─────────────┤
│  utils/     │  ← 基础工具层：零依赖的纯函数
└─────────────┘
```

- **依赖方向向下**：上层可调用下层，下层不依赖上层
- **每层可独立测试**：utils 和 analytics 是纯函数/纯逻辑，无需 mock；db 和 external 层可通过 mock 隔离
- **职责单一**：每个模块只做一件事，修改时影响范围可控
