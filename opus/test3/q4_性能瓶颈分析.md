# 问题4：分析 get_dashboard 接口性能瓶颈

## 原始伪代码

```python
def get_dashboard(user_id):
    user = db.query('SELECT * FROM users WHERE id = %s', user_id)
    orders = db.query('SELECT * FROM orders WHERE user_id = %s', user_id)
    for order in orders:
        order['items'] = db.query('SELECT * FROM items WHERE order_id = %s', order['id'])
        order['payment'] = db.query('SELECT * FROM payments WHERE order_id = %s', order['id'])
    notifications = requests.get(f'https://api.internal/notifications/{user_id}').json()
    return {'user': user, 'orders': orders, 'notifications': notifications}
```

## 瓶颈分析

### 瓶颈1：N+1 查询问题（最严重 🔴）

**问题**：`for order in orders` 循环内部对每个订单执行 2 次数据库查询（items + payments）。如果用户有 100 个订单，总共要执行 **2 + 100×2 = 202 次** 数据库查询。

**影响**：数据库查询数量随订单数线性增长，每次查询都有网络往返开销（RTT），大量订单时响应时间急剧增加。

**优化思路**：
- 将循环内的单条查询改为**批量查询**：一次性查出所有订单的 items 和 payments，在内存中用 `order_id` 做分组关联
- 或者使用 SQL JOIN 在一条查询中关联 orders + items + payments

### 瓶颈2：串行执行无并发（🟡）

**问题**：三个独立的数据源（user 查询、orders 查询、notifications HTTP 请求）是**顺序串行**执行的。特别是 `notifications` 是外部 HTTP 调用，延迟不可控。

**影响**：总耗时 = user 查询时间 + orders 查询时间 + N+1 循环时间 + notifications HTTP 时间。即使各自都很快，串行叠加后也很慢。

**优化思路**：
- 用 `asyncio` 或 `concurrent.futures` 将 **互不依赖** 的查询并行执行
- user、orders、notifications 三者可以并行；items/payments 在 orders 返回后并行批量查询

### 瓶颈3：`SELECT *` 全列查询（🟡）

**问题**：所有查询都用 `SELECT *`，获取表中全部字段。

**影响**：
- 传输了不需要的列，浪费带宽和内存
- 无法利用覆盖索引（covering index）优化
- 如果表有 BLOB/TEXT 大字段，性能影响更大

**优化思路**：显式指定需要的列，如 `SELECT id, name, email FROM users`

### 瓶颈4：无缓存层（🟡）

**问题**：每次请求都直接查数据库和调外部 API，没有缓存。

**影响**：Dashboard 是高频访问页面，同一用户短时间内多次访问会重复执行相同的查询。

**优化思路**：
- 对变化频率低的数据（user 信息）加 Redis 缓存
- 对 notifications 加短时 TTL 缓存（如 30 秒）
- orders 数据可以按最后更新时间做条件缓存

### 瓶颈5：外部 API 调用无超时和降级（🟠）

**问题**：`requests.get(...)` 没有设置超时，如果通知服务不可用，整个 dashboard 接口会被阻塞甚至挂起。

**优化思路**：
- 设置合理的超时（如 3 秒）
- 添加降级策略：通知服务不可用时返回空列表，不影响主数据展示
- 考虑用断路器模式（Circuit Breaker）

## 优化优先级总结

| 优先级 | 瓶颈 | 预期收益 |
|--------|------|----------|
| P0 | N+1 查询 → 批量查询 | 202 次查询降为 4 次，数量级提升 |
| P1 | 串行 → 并发执行 | 总耗时从累加变为取最大值 |
| P1 | 外部 API 加超时和降级 | 防止级联故障 |
| P2 | `SELECT *` → 指定列 | 减少传输量，利用索引 |
| P2 | 加缓存层 | 减少重复查询，降低数据库压力 |
