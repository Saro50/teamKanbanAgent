## 性能瓶颈分析

### 1. 最严重：N+1 查询问题
```
查询次数 = 1 (user) + 1 (orders) + N×2 (items + payments)
```
如果用户有 100 个订单，就会执行 201 次数据库查询，这是典型的 N+1 问题。

### 2. SELECT * 问题
每个查询都获取所有字段，可能包含不需要的数据，增加网络传输和内存开销。

### 3. 串行执行
所有查询按顺序执行，没有利用并发：
```
user → orders → [order1查询, order2查询, ...] → notifications
```

### 4. 外部 API 无超时控制
`requests.get()` 没有设置超时，可能长时间阻塞。

### 5. 缺少缓存
重复查询相同用户的热点数据。

---

## 优化方案

### 方案一：批量查询替代循环查询（最核心）
```sql
-- 一次性获取所有 items
SELECT * FROM items WHERE order_id IN (order_id1, order_id2, ...)

-- 一次性获取所有 payments
SELECT * FROM payments WHERE order_id IN (order_id1, order_id2, ...)
```
**效果：** 查询次数从 1+1+N×2 降到 4，与订单数量无关。

### 方案二：只查询必需字段
```sql
SELECT id, name, email FROM users WHERE id = %s
```

### 方案三：并发执行独立查询
数据库查询（user + orders + items + payments）可以并发执行，与 notifications API 调用也可以并发。

### 方案四：添加 API 超时和重试
```python
requests.get(url, timeout=(3, 10), retries=2)
```

### 方案五：引入多级缓存
- L1: 用户信息缓存（Redis）
- L2: Dashboard 数据缓存（5-10 分钟）
- L3: 外部 API 结果缓存

### 方案六：数据结构优化
考虑使用 JOIN 一次性获取关联数据，减少应用层组装。

### 方案七：分页/懒加载
如果订单很多，考虑分页返回或只返回最近 N 条订单。

## 优先级排序
1. **批量查询** → 立竿减少 90%+ 查询次数
2. **外部 API 优化** → 防止超时累积
3. **并发执行** → 进一步减少总响应时间

```python
        'SELECT order_id, amount, status, payment_method FROM payments WHERE order_id IN %s',
        (tuple(order_ids),)
    )
    
    # 按order_id索引（每个订单只有一个支付记录）
    payments_map = {p['order_id']: p for p in payments}
    return payments_map

async def get_notifications(user_id: int) -> List[Dict]:
    """获取通知信息（带超时和重试）"""
    timeout = aiohttp.ClientTimeout(total=5, connect=2)
    
    async with aiohttp.ClientSession(timeout=timeout) as session:
        try:
            async with session.get(f'https://api.internal/notifications/{user_id}') as response:
                if response.status == 200:
                    return await response.json()
                return []
        except (aiohttp.ClientError, asyncio.TimeoutError):
            return []  # 外部服务不可用时返回空列表，不阻塞主流程
```

## 主要优化点：

1. **异步并发**：所有独立查询并行执行
2. **批量查询**：N+1 问题彻底解决，查询次数固定为 5 次
3. **多级缓存**：用户信息内存缓存 + dashboard Redis 缓存
4. **字段精简**：只查询必需字段
5. **超时控制**：外部 API 设置合理超时
6. **错误隔离**：外部服务故障不影响核心功能
7. **分页限制**：限制订单数量防止数据爆炸

## 性能提升估算：
- 原方案：1 + 1 + N×2 + 1 次查询（串行）
- 优化后：5 次查询（并行） + 缓存命中
- 响应时间：从 O(N) 降到 O(1)，并发执行进一步减少总耗时

```python
class Processor:
    """处理数据集，计算满足条件的统计信息"""
    
    def __init__(self, data: List[Dict[str, Any]]) -> None:
        """
        初始化处理器
        
        Args:
            data: 数据列表，每个元素是包含 's' 和 'v' 键的字典
        """
        self.data = data
    
    def calculate_statistics(self) -> Dict[str, Any]:
        """
        计算满足条件的数据项的统计信息
        
        条件：状态为 'a' 且值大于 0
        
        Returns:
            包含 'avg'（平均值）、'count'（计数）、'total'（总和）的字典
            如果没有满足条件的数据，返回 0
        """
        filtered_items = self._filter_active_positive_items()
        
        if not filtered_items:
            return 0
        
        return self._compute_statistics(filtered_items)
    
    def _filter_active_positive_items(self) -> List[Dict[str, Any]]:
        """过滤出状态为 'a' 且值大于 0 的项目"""
        return [
            item for item in self.data 
            if item.get('s') == 'a' and item.get('v', 0) > 0
        ]
    
    def _compute_statistics(self, items: List[Dict[str, Any]]) -> Dict[str, Any]:
        """计算项目的统计信息"""
        total = sum(item['v'] for item in items)
        count = len(items)

            'avg': total / count,
            'count': count,
```
