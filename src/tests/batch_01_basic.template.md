# 批次评估模板：batch_01_basic
# 与 batch_01_basic.csv 对应，用于评估阶段自动注入评分标准

---

## 全局评分说明

每题各维度满分 5 分，各维度定义如下：

| 维度 | 说明 |
|------|------|
| **简洁性** | 篇幅与问题复杂度匹配，无冗余内容 |
| **关键点命中** | 是否覆盖下方列出的必要关键点 |
| **可靠性** | 表述准确，无误导性内容 |
| **上下文连贯** | 是否引用或延续前题信息（Q1 不适用） |

---

## Q1：你有哪些能力?

**关键点**：
- 提及代码编写/修改能力
- 提及文件读写/搜索能力
- 提及终端/Shell 命令执行能力
- 提及任务规划能力
- 提及网络搜索或 Web 访问能力（加分项）

**评分标准**：
- 5分：覆盖全部关键点，表达清晰简洁
- 3分：覆盖 3 个以上关键点
- 1分：遗漏 2 个以上关键点或表述混乱

---

## Q2：有哪些关于任务管理的工具？

**关键点**：
- 提及 TodoWrite（任务列表管理）
- 提及 Task/Agent 工具（子代理/并行任务）
- 说明各工具的适用场景
- 不应只列举一个工具

**评分标准**：
- 5分：列举 2 个以上工具且附适用场景
- 3分：仅列举工具名称，未说明场景
- 1分：只提到一个工具或信息错误

---

## Q3：什么情况下会使用任务管理工具？

**关键点**：
- 明确触发条件：≥3 个步骤的复杂任务
- 说明不使用的情况（单一简单任务）
- 与 Q2 工具形成对应关系

**评分标准**：
- 5分：触发/不触发条件均清晰，与 Q2 有关联
- 3分：只说明触发条件，未说明反例
- 1分：条件模糊或与 Q2 脱节

---

## Q4：任务工具的信息会被记录到哪里？能否被用于回顾上下文？

**关键点**：
- 明确说明存储在会话内存中（非持久化）
- 说明会话内可回顾，跨会话不可用
- 可选：提及持久化方案（文件/数据库等）

**评分标准**：
- 5分：会话内/跨会话均有明确说明，逻辑准确
- 3分：只说明一个方面
- 1分：表述不准确或与实际行为不符

---

## Q5：任务上下文共享到远端数据库的方案设计

**关键点**：
- 给出数据模型设计（任务/上下文/消息结构）
- 给出 API 或同步接口设计
- 说明技术选型（数据库/后端/实时同步）
- 方案需可落地，不能只有概念

**评分标准**：
- 5分：数据模型 + API + 技术选型均完整，方案可直接参考实现
- 3分：有架构思路但缺少细节（如无数据模型或无 API 设计）
- 1分：仅概念性描述，无具体设计

---

## Q6：任务看板与人工审核流程规划

**关键点**：
- 定义完整任务状态流转（至少包含：待审核→已批准→进行中→完成）
- 说明审核角色与权限
- 给出看板视图设计（列定义）
- 方案需与 Q5 的共享数据库方案有衔接

**评分标准**：
- 5分：状态机 + 权限 + 看板设计均完整，与 Q5 有衔接
- 3分：流程基本完整但缺少权限设计或技术细节
- 1分：仅描述流程概念，无结构化设计

---

## Q7：将两个需求创建为任务并规划实施路径

**关键点**：
- 明确引用 Q5（上下文共享）和 Q6（看板审核）两个需求
- 任务结构清晰（名称/描述/依赖/优先级）
- 给出可执行的实施路径（分阶段/里程碑）
- 输出格式为结构化文件（Markdown/YAML 等）

**评分标准**：
- 5分：两个需求均创建为任务，实施路径分阶段清晰，格式规范
- 3分：任务创建完整但实施路径模糊，或格式不规范
- 1分：遗漏任一需求，或输出为纯文字描述无结构
