## 全局评分说明

每题各维度满分 5 分，各维度定义如下：

| 维度 | 说明 |
|------|------|
| **正确性** | 代码能正确运行或分析结论准确无误 |
| **工程质量** | 代码风格、命名、结构是否符合工程实践标准 |
| **完整性** | 是否覆盖了题目要求的所有点，无遗漏 |
| **上下文连贯** | 是否引用或延续前题的代码和结论（Q1 不适用） |

## Q1：阅读代码找 bug（只列问题不给修复）

**设计意图**：纯代码阅读能力——能否逐函数精准识别缺陷，且遵守"不给修复代码"的约束。

**关键点**：
- `merge_dicts`：`d.items()` 误写为 `d`（遍历 dict 直接迭代只拿到 key，解包会报错）
- `get_user`：`return None` 缩进错误，在第一次循环就返回，只能检查第一个 user
- `safe_divide`：`except TypeError` 分支无 return，会隐式返回 None
- 不应包含修复代码（遵守约束）
- 可选加分：指出 `safe_divide` 返回 0 可能掩盖真实错误（设计隐患）

**评分标准**：
- 5分：三个 bug 全部准确识别 + 无修复代码 + 描述清晰
- 3分：识别 2 个 bug，或包含了修复代码违反约束
- 1分：遗漏 2 个以上 bug，或分析有误

## Q2：给出修复代码并附注释

**设计意图**：从"只读"切换到"写"——验证修复是否精准对应 Q1 列出的问题，且注释简洁准确。

**关键点**：
- `merge_dicts`：将 `for k, v in d:` 修复为 `for k, v in d.items():`
- `get_user`：将 `return None` 退回到 for 循环外层
- `safe_divide`：在 `except TypeError` 中增加 return 语句（如 return None）
- 每个函数附一行注释说明修复点
- 代码能正确运行

**评分标准**：
- 5分：三处修复全部正确 + 每个附注释 + 代码可直接运行
- 3分：修复正确但注释缺失，或修复了但引入了新问题
- 1分：修复不正确或遗漏关键修复项

## Q3：编写 pytest 单元测试

**设计意图**：测试编写能力——验证 Agent 能否为自己写的代码设计覆盖全面的测试用例。

**关键点**：
- 使用 pytest 框架（非 unittest）
- 每个函数至少 3 个用例：正常路径 / 边界条件 / 异常路径
  - `merge_dicts`：正常合并 / 空字典 / override=False 场景
  - `get_user`：找到用户 / 空列表或未找到 / user_id 类型边界
  - `safe_divide`：正常除法 / 除以零 / 类型错误输入
- 测试命名清晰（如 `test_merge_dicts_empty_input`）
- 使用 Q2 修复后的函数（上下文衔接）

**评分标准**：
- 5分：9+ 个测试用例（3函数 × 3路径），命名规范，覆盖完整，引用 Q2 代码
- 3分：测试数量不足（<9）或缺少某一路径类型，但基本结构正确
- 1分：测试用例 < 5 个，或使用了错误的测试框架，或与 Q2 代码不匹配

## Q4：分析性能瓶颈（思路，不写完整代码）

**设计意图**：工程诊断能力——面对典型的 N+1 查询 + 同步外部调用场景，能否准确识别瓶颈并给出系统化优化思路。

**关键点**：
- 识别 N+1 查询问题（orders 循环内逐条查 items 和 payments）
- 识别 `SELECT *` 可能拉取不必要字段
- 识别同步外部 HTTP 调用（notifications）阻塞主流程
- 优化思路应包含：
  - 批量查询 / JOIN 替代逐条查询
  - 并发或异步处理外部调用
  - 可选：缓存层、只查必要字段
- 以思路为主，不需要完整代码

**评分标准**：
- 5分：N+1 + SELECT * + 同步阻塞三大问题均识别，优化方案具体可落地
- 3分：识别出 N+1 但遗漏其他问题，或方案过于笼统
- 1分：未识别出 N+1 问题，或给出了不相关的优化建议

## Q5：重写优化后的代码

**设计意图**：从思路到代码的落地——验证 Agent 能否将 Q4 的分析转化为可运行的优化代码，且保持接口兼容。

**关键点**：
- 接口返回结构 `{'user': ..., 'orders': ..., 'notifications': ...}` 不变
- items/payments 使用批量查询（WHERE order_id IN (...)）
- notifications 调用与数据库查询并发（如 asyncio / threading / concurrent.futures）
- 代码风格与 Q4 方案保持一致
- 可选加分：加入错误处理、超时设置

**评分标准**：
- 5分：返回结构不变 + 批量查询 + 并发 + 代码可读 + 与 Q4 方案对应
- 3分：实现了部分优化但不完整（如只做了批量查询未做并发），或与 Q4 思路不一致
- 1分：返回结构被改变，或代码有明显 bug，或完全未反映 Q4 的优化点

## Q6：混淆代码重构

**设计意图**：重构能力的极端测试——面对极度压缩的代码（单字母命名、一行多语句），能否准确理解语义并重构为高质量代码。

**关键点**：
- 正确理解原代码语义：筛选 status='a' 且 value>0 的记录，计算平均值/计数/总计
- 类名和变量名重命名为有意义的名称（如 `P` → `DataProcessor`/`MetricsCalculator`）
- 方法拆分合理（单一职责）
- 保持外部行为不变（返回值结构和逻辑一致）
- 可选加分：增加类型注解、docstring

**评分标准**：
- 5分：语义完全理解 + 命名优良 + 结构清晰 + 行为不变 + 有类型注解/文档
- 3分：理解正确、行为不变，但命名或结构改进有限
- 1分：误解原代码逻辑，或重构后行为发生变化

## Q7：目录结构与模块划分建议

**设计意图**：架构设计能力——验证 Agent 能否从散落的代码片段中抽象出合理的项目结构，体现全局工程思维。

**关键点**：
- 给出树形目录结构（使用标准 tree 格式）
- 覆盖本次对话中涉及的代码：工具函数（Q2）、数据查询（Q5）、数据处理（Q6）、测试（Q3）
- 模块划分合理（如 utils / services / models / tests）
- 每个模块有简要职责说明
- 整体结构符合 Python 项目惯例（如包含 `__init__.py`、`requirements.txt` 等）

**评分标准**：
- 5分：树形结构完整 + 涵盖所有对话代码 + 模块划分合理 + 职责说明清晰
- 3分：结构基本合理但遗漏部分代码，或职责说明模糊
- 1分：结构混乱或与对话内容无关，未体现模块化思维
